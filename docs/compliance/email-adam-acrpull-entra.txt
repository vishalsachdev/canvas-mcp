Hi Adam,

Two small things on the hosted Canvas MCP app (gies-canvas-mcp-staging), both in your wheelhouse as a subscription Owner since the rest of us are Contributor-only:

1) AcrPull role grant — retires a security workaround

The app currently pulls its container image using ACR admin-user credentials, only because its managed identity was never granted AcrPull (and granting roles needs an Owner). If you grant the app's managed identity AcrPull on the registry, we can turn the admin-user account off — cleaner and more secure. One command does it:

az role assignment create \
  --assignee dd6ad3e0-e1af-4845-b259-f71e1e2c1dcf \
  --role AcrPull \
  --scope $(az acr show -n giescanvasmcpacr -g DL_ResourceGroup_01 --query id -o tsv)

(The assignee GUID is the app's managed-identity object id; registry is giescanvasmcpacr in DL_ResourceGroup_01.)

2) Heads-up: per-user NetID sign-in for the MCP app — no action needed

To handle student data properly under FERPA (each user authenticated as themselves rather than sharing one key), we're adding University NetID/Entra sign-in to the app. We'll self-register two app registrations in the tenant — an API plus a desktop client — the same self-serve path uniquick and illinihunt used, and we don't need admin consent (we pre-authorize our own client). Just flagging it so you're aware, and in case there's a tenant policy or naming convention you'd want us to follow. If you'd prefer we route the registrations through Technology Services instead, just say so.
